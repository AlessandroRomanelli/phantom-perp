# Phase 9.1: Coinbase Advanced API Migration - Research

**Researched:** 2026-03-23
**Domain:** REST API migration (Coinbase INTX to Advanced Trade), JWT authentication, endpoint remapping
**Confidence:** HIGH

## Summary

This phase migrates all 10 REST API calls from Coinbase INTX (`api.international.coinbase.com/api/v1`) to the Coinbase Advanced Trade API (`api.coinbase.com/api/v3/brokerage`). The migration involves three intertwined changes: (1) switching the base URL and all endpoint paths, (2) replacing HMAC-SHA256 auth with ES256 JWT auth via Cloud API Keys, and (3) adapting Pydantic response models to match the new field names/nesting. The WebSocket client is already on Advanced Trade URLs and requires no changes.

The Advanced Trade API uses a fundamentally different structure: endpoints are grouped by resource type (`/products/`, `/orders/`, `/intx/`) rather than the INTX pattern (`/instruments/`, `/positions/`). Perpetuals-specific data (positions, portfolio summary) lives under the `/intx/` prefix and requires a `portfolio_uuid` path parameter. The product ID format for perpetual contracts is `XXX-PERP-INTX` (e.g., `ETH-PERP-INTX`), which the WS client already uses.

**Primary recommendation:** Rewrite `auth.py` to generate ES256 JWTs, rewrite `rest_client.py` to use `/api/v3/brokerage/` paths, update all Pydantic models in `models.py` to match new response shapes. Add `PyJWT` and `cryptography` as dependencies.

### Endpoint Mapping (INTX to Advanced Trade)

This is the critical reference table. Each row maps a current INTX endpoint to its Advanced Trade equivalent.

| # | Current Method | INTX Path | Adv Trade Path | HTTP | Key Differences |
|---|---------------|-----------|----------------|------|-----------------|
| 1 | `get_instruments()` | `/api/v1/instruments` | `/api/v3/brokerage/products` | GET | Filter by `product_type=FUTURE`, `contract_expiry_type=PERPETUAL`. Response wraps in `{"products": [...]}` |
| 2 | `get_orderbook()` | `/api/v1/instruments/{id}/book` | `/api/v3/brokerage/product_book?product_id={id}` | GET | Product ID is query param, not path. Response wraps in `{"pricebook": {...}}`. Levels are `{price, size}` (no `num_orders`) |

| 4 | `get_funding_rate()` | `/api/v1/instruments/{id}/funding` | `/api/v3/brokerage/products/{id}` | GET | No dedicated funding endpoint. Funding rate lives in `future_product_details.perpetual_details.funding_rate` of the get_product response |
| 5 | `get_portfolio()` | `/api/v1/portfolios` | `/api/v3/brokerage/intx/portfolio/{portfolio_uuid}` | GET | Requires portfolio_uuid as path param. Response has `collateral`, `unrealized_pnl`, `total_balance`, `portfolio_initial_margin`, etc. |

| 7 | `get_open_orders()` | `/api/v1/orders?status=OPEN` | `/api/v3/brokerage/orders/historical/batch?order_status=OPEN` | GET | Path is `historical/batch` not just `orders`. Response: `{"orders": [...]}` |

### Auth Pattern: JWT Generation

```python
# Source: Coinbase official Python SDK jwt_generator.py
import secrets
import time

import jwt
from cryptography.hazmat.primitives import serialization


class CoinbaseAuth:
    """JWT-based auth for Coinbase Advanced Trade API.

    Args:
        api_key: Cloud API Key name (UUID).
        api_secret: PEM-encoded EC private key string.
    """

    def __init__(self, api_key: str, api_secret: str) -> None:
        self.api_key = api_key
        self._private_key = serialization.load_pem_private_key(
            api_secret.encode("utf-8"), password=None
        )

    def sign(self, method: str, path: str, body: str = "") -> dict[str, str]:
        """Generate Authorization header with JWT for a REST request.

        Args:
            method: HTTP method (GET, POST, etc.).
            path: Request path (e.g., '/api/v3/brokerage/orders').
            body: Unused for JWT (kept for interface compat).

        Returns:
            Headers dict with Authorization bearer token.
        """
        uri = f"{method.upper()} api.coinbase.com{path}"
        jwt_payload = {
            "sub": self.api_key,
            "iss": "cdp",
            "nbf": int(time.time()),
            "exp": int(time.time()) + 120,
            "uri": uri,
        }
        token = jwt.encode(
            jwt_payload,
            self._private_key,
            algorithm="ES256",
            headers={"kid": self.api_key, "nonce": secrets.token_hex()},
        )
        return {"Authorization": f"Bearer {token}"}
```

**Key details:**
- Algorithm: ES256 (ECDSA with P-256 curve, NOT Ed25519)
- Token lifetime: 120 seconds (2 minutes)
- Each request needs a fresh JWT (the `uri` claim embeds method + path)
- No caching needed -- JWT generation is ~1ms with EC key already loaded
- The PEM private key is loaded once in `__init__`, reused for all sign calls
- `nonce` is a random hex string for replay protection

### Recommended Discretion: JWT Caching Strategy

**Recommendation: Do NOT cache JWTs.** Each JWT embeds the specific `uri` (method + path), so tokens for different endpoints are not interchangeable. JWT generation with an already-loaded EC key takes <1ms. The complexity of a cache (keyed by method+path, with 90s TTL to stay under 120s expiry) is not worth it.

### Recommended Discretion: Error Code Mapping

The Advanced Trade API returns errors differently:
- **INTX:** HTTP status codes (400, 422, 429) with error message in response body text
- **Advanced Trade:** Similar HTTP codes, but order creation wraps errors in `{"success": false, "error_response": {"message": "...", "new_order_failure_reason": "..."}}`

**Recommendation:** Keep the existing error detection pattern in `_request()` (check status code >= 400, check for "insufficient margin" in body). For `create_order()`, also check the `success` boolean in the response and extract `error_response.message` on failure.

### Recommended Discretion: Rate Limiter Headers

The Advanced Trade API uses standard rate limit headers. The existing `RateLimiter.update_from_headers()` already reads `RateLimit-Remaining` and `RateLimit-Reset`. **Recommendation: Keep as-is.** If the Advanced Trade API uses different header names, the rate limiter gracefully degrades to its internal token bucket (no headers = no update, bucket refills naturally).

### Portfolio UUID Requirement

The Advanced Trade API perpetuals endpoints (`/intx/positions/`, `/intx/portfolio/`) require `portfolio_uuid` as a path parameter. The current INTX endpoints do NOT require this because the API key implicitly scopes to one portfolio.

**Impact:** `CoinbaseRESTClient` needs to know its portfolio UUID. The `PortfolioIDSettings` already has `portfolio_a_id` configured. The client constructor should accept a `portfolio_uuid` parameter, stored as an instance attribute.

### Product ID Resolution at Startup

Per D-14 through D-17, product IDs must be discovered dynamically:

1. Call `GET /api/v3/brokerage/products?product_type=FUTURE&contract_expiry_type=PERPETUAL`
2. Filter results where `product_venue == "INTX"` (for INTX perps specifically)
3. Match `base_display_symbol` or `base_currency_id` against configured symbols (ETH, BTC, SOL, QQQ, SPY)
4. Store mapping: `{"ETH": "ETH-PERP-INTX", "BTC": "BTC-PERP-INTX", ...}`
5. Update `InstrumentConfig` registry with resolved product IDs

The `ws_product_id` property on `InstrumentConfig` already constructs `f"{self.id}-INTX"` -- but with dynamic resolution, the instrument `id` field itself should store the resolved product ID (e.g., `ETH-PERP-INTX`), or a separate `product_id` field is added.

### Response Model Transformations

The biggest model changes:

**Positions:** Fields become `Amount` objects (`{value, currency}`) instead of plain `Decimal`:
```python
# INTX: unrealized_pnl: Decimal
# Advanced Trade: unrealized_pnl: {"value": "123.45", "currency": "USDC"}
```

**Candles:** Field values become strings, field order differs:
```python
# INTX: start: datetime, open: Decimal, high: Decimal, ...
# Advanced Trade: start: str (unix ts), open: str, high: str, ...
```

**Orders:** Creation response is completely restructured:
```python
# INTX: returns full order object
# Advanced Trade: returns {"success": true, "success_response": {"order_id": "...", ...}}
```

**Fills:** Field name changes: `fee` -> `commission`, `filled_at` -> `trade_time`, `fill_id` -> `entry_id`

### Anti-Patterns to Avoid
- **Using coinbase-advanced-py SDK as a dependency:** Adds a heavy SDK when we only need JWT generation. Our existing httpx client is more controllable.
- **Caching JWTs:** The `uri` claim makes each JWT endpoint-specific. Not worth the complexity.
- **Hardcoding product IDs:** The product ID format could change; resolve dynamically per D-14.
- **Mixing old and new auth:** Big-bang swap means no intermediate state. Don't try to support both auth methods simultaneously.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JWT generation | Custom ECDSA signing | PyJWT + cryptography | EC key handling has subtle pitfalls (key format parsing, DER vs PEM, curve validation) |
| PEM key parsing | Manual base64/ASN.1 parsing | cryptography.hazmat.primitives.serialization | PEM format has variants; library handles all |

**Key insight:** The auth layer is the only truly new code. Everything else is endpoint path + model field remapping.

## Common Pitfalls

### Pitfall 1: JWT URI Format
**What goes wrong:** JWT rejected because `uri` claim doesn't match the request.
**Why it happens:** The `uri` must be exactly `"{METHOD} api.coinbase.com{path}"` (no `https://`, no trailing slash, method uppercase).
**How to avoid:** Use the exact format from Coinbase SDK: `f"{method.upper()} api.coinbase.com{path}"`
**Warning signs:** 401 Unauthorized on every request.

### Pitfall 2: ES256 vs Ed25519 Key Type
**What goes wrong:** JWT signing fails with cryptography error.
**Why it happens:** Cloud API Key portal can generate Ed25519 keys, but Advanced Trade only supports ES256 (EC P-256).
**How to avoid:** Document that users must generate ES256 keys. Validate key type at startup.
**Warning signs:** `UnsupportedAlgorithm` exception from PyJWT.

### Pitfall 3: Amount Objects in Position/Portfolio Responses
**What goes wrong:** Pydantic validation fails because `unrealized_pnl` is now `{"value": "123", "currency": "USDC"}` not `"123"`.
**Why it happens:** Advanced Trade API uses `Amount` objects for monetary values in perpetuals endpoints.
**How to avoid:** Create an `Amount` Pydantic model, add validators to extract `Decimal(value)` from nested objects.
**Warning signs:** `ValidationError` on position/portfolio responses.

### Pitfall 4: Candle Timestamps as UNIX Strings
**What goes wrong:** Candle start times parsed incorrectly.
**Why it happens:** INTX returns ISO 8601 datetimes; Advanced Trade returns UNIX timestamps as strings.
**How to avoid:** Add a validator that converts string UNIX timestamps to datetime.
**Warning signs:** Wrong candle times, off-by-hours errors.

### Pitfall 5: Create Order Response Structure
**What goes wrong:** Order creation appears to fail when it succeeded.
**Why it happens:** Advanced Trade wraps the response in `{"success": true, "success_response": {...}}`. Code looking for top-level `order_id` won't find it.
**How to avoid:** Check `response["success"]` first, then extract from `success_response`.
**Warning signs:** All orders "failing" despite appearing on exchange.

### Pitfall 6: Cancel Order Changed to POST
**What goes wrong:** Cancel returns 404 or method not allowed.
**Why it happens:** INTX uses `DELETE /orders/{id}`; Advanced Trade uses `POST /orders/batch_cancel` with body `{"order_ids": ["id"]}`.
**How to avoid:** Change cancel_order to use POST with body.
**Warning signs:** 404 or 405 on cancel attempts.

### Pitfall 7: Portfolio UUID Required for Perpetuals Endpoints
**What goes wrong:** 400/404 on position and portfolio queries.
**Why it happens:** `/intx/positions/{portfolio_uuid}` requires the UUID in the path; INTX inferred it from the API key.
**How to avoid:** Store portfolio_uuid in CoinbaseRESTClient, inject in path for perpetuals endpoints.
**Warning signs:** Missing path parameter errors.

### Pitfall 8: Funding Rate Not a Separate Endpoint
**What goes wrong:** 404 on funding rate requests.
**Why it happens:** Advanced Trade has no `/instruments/{id}/funding` equivalent. Funding rate is embedded in the `get_product` response under `future_product_details.perpetual_details.funding_rate`.
**How to avoid:** Rewrite `get_funding_rate()` to call `get_product()` and extract nested funding data.
**Warning signs:** Endpoint not found errors for funding.

### Product ID Discovery
```python
async def discover_product_ids(client: CoinbaseRESTClient, symbols: list[str]) -> dict[str, str]:
    """Resolve canonical symbols to Advanced Trade product IDs.

    Args:
        client: Authenticated REST client.
        symbols: Base currency symbols (e.g., ["ETH", "BTC", "SOL"]).

    Returns:
        Mapping of symbol to product_id (e.g., {"ETH": "ETH-PERP-INTX"}).
    """
    products = await client.get_products(
        product_type="FUTURE",
        contract_expiry_type="PERPETUAL",
    )
    mapping: dict[str, str] = {}
    for product in products:
        if product.product_venue != "INTX":
            continue
        for symbol in symbols:
            if product.base_display_symbol == symbol or product.base_currency_id == symbol:
                mapping[symbol] = product.product_id
                break
    return mapping
```

## Open Questions

1. **Rate limit headers on Advanced Trade**
   - What we know: INTX uses `RateLimit-Remaining` and `RateLimit-Reset`. Advanced Trade likely uses similar headers but this is not explicitly documented.
   - What's unclear: Exact header names and rate limit budget per key.
   - Recommendation: Keep existing `update_from_headers()` logic. If headers are absent, the internal token bucket handles throttling. Monitor logs after deployment.

2. **Product ID stability for QQQ-PERP-INTX and SPY-PERP-INTX**
   - What we know: ETH-PERP-INTX, BTC-PERP-INTX, SOL-PERP-INTX are well-established. QQQ and SPY perpetuals are newer.
   - What's unclear: Whether these products consistently appear in the products list, or have different venue/naming.
   - Recommendation: Dynamic discovery handles this -- if a product is missing, log a warning and skip that instrument rather than crashing.

3. **PEM key format in environment variables**
   - What we know: PEM keys contain newlines. Environment variables typically don't have literal newlines.
   - What's unclear: Whether users will use `\n` escape sequences or actual multiline env var values.
   - Recommendation: Add a normalizer that replaces literal `\n` strings with actual newlines before passing to `load_pem_private_key()`. This is a common pattern for PEM keys in env vars.
