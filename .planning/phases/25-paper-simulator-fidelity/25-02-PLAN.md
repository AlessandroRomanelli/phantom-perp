<objective>
Add stop-loss slippage to the paper simulator's protective order monitor.

Purpose: Stop-loss orders in real markets rarely fill at the exact trigger price — fast-market conditions cause slippage of 0.05-0.15%. Paper mode currently fills SL orders at the exact trigger price, making paper P&L overly optimistic on losing trades. Adding configurable slippage produces more realistic drawdown metrics.

Output: SL slippage logic in `protective_order_monitor`, with TDD test coverage, wired to the `PaperSimulatorConfig.sl_slippage_bps` parameter created in Plan 01.
</objective>

   ```python
   def _apply_sl_slippage(
       fill_price: Decimal,
       side: OrderSide,
       is_stop_loss: bool,
       cfg: PaperSimulatorConfig,
   ) -> Decimal:
       """Apply slippage to stop-loss fill price.

       SL slippage models fast-market execution degradation. Direction is always
       adverse to the trader: BUY-to-close (closing short) fills higher,
       SELL-to-close (closing long) fills lower.

       Take-profit orders are returned unchanged.
       """
       if not is_stop_loss or cfg.sl_slippage_bps == 0:
           return fill_price

       bps = cfg.sl_slippage_bps / Decimal("10000")
       if side == OrderSide.BUY:
           slipped = fill_price * (Decimal("1") + bps)
       else:
           slipped = fill_price * (Decimal("1") - bps)

       return slipped.quantize(Decimal("0.01"), rounding=ROUND_HALF_UP)
   ```

2. Wire into `protective_order_monitor` inner task. In the triggered-order loop (around line 798), BEFORE the `portfolio.apply_fill()` call, compute the slipped price:
