# Phase 9.1: Coinbase Advanced API Migration - Context

**Gathered:** 2026-03-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrate all REST API calls from Coinbase INTX (`api.international.coinbase.com/api/v1`) to Coinbase Advanced Trade API. This is a 1:1 migration of existing functionality — no new endpoints or capabilities. WebSocket is already on Advanced Trade URLs and working; only REST and auth need changes. This is blocking v1.1 completion.

</domain>

<decisions>
## Implementation Decisions

### Migration Scope
- **D-01:** All 10 REST endpoints migrate to Advanced Trade equivalents (market data, orders, positions, portfolio, fills)
- **D-02:** WebSocket is already working on Advanced Trade URLs — no WS changes needed
- **D-03:** 1:1 endpoint migration only — no new endpoints or capabilities added
- **D-04:** This phase blocks v1.1 milestone completion

### Authentication
- **D-05:** Switch from HMAC-SHA256 to Cloud API Keys with JWT authentication
- **D-06:** Key material provided as key name (UUID) + PEM secret, both in environment variables
- **D-07:** Env vars renamed from `COINBASE_INTX_*` to `COINBASE_ADV_*` prefix
- **D-08:** New env vars: `COINBASE_ADV_API_KEY_A/B` (key name), `COINBASE_ADV_API_SECRET_A/B` (PEM secret)
- **D-09:** Passphrase env vars (`COINBASE_INTX_PASSPHRASE_A/B`) dropped — not needed for JWT

### Cutover Strategy
- **D-10:** Big-bang swap — rewrite `rest_client.py` and `auth.py` together, all endpoints switch at once
- **D-11:** No rollback mechanism needed — one-way migration, git history is the rollback
- **D-12:** Clean up stale INTX URLs in `constants.py` (DEFAULT_WS_MARKET_URL, DEFAULT_WS_USER_URL currently point to INTX but are unused)
- **D-13:** Advanced Trade REST base URL hardcoded as a constant (not configurable via env var)

### Product ID Resolution
- **D-14:** Dynamic product ID discovery at startup via instruments endpoint — not hardcoded
- **D-15:** Filter instruments by type (perpetual/future) and match by base currency (ETH, BTC, SOL, QQQ, SPY)
- **D-16:** YAML config stores canonical symbols (ETH, BTC, SOL, QQQ, SPY); actual product IDs resolved at startup
- **D-17:** Resolved product IDs used for WS subscriptions, REST queries, and order placement

### Claude's Discretion
- JWT token caching and refresh strategy (short-lived tokens need periodic regeneration)
- Advanced Trade API response model field mapping (researcher to determine exact field names)
- Error code mapping between INTX and Advanced Trade error responses
- Rate limiter adjustments if Advanced Trade has different rate limit headers

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `CoinbaseRESTClient` class structure — method signatures and return types stay the same, only internals change
- `CoinbaseClientPool` — portfolio routing via `PortfolioTarget` enum is API-agnostic, reuse as-is
- `RateLimiter` — token-bucket implementation is reusable, may need header name updates
- All Pydantic response models — field names may change but model pattern stays

### Established Patterns
- All REST calls go through `_request()` private method in rest_client.py — single point of change for URL/auth
- Auth is injected via `_sign_request()` — clean replacement point for JWT
- WS auth uses separate `generate_ws_signature()` — already works, don't touch
- Config loading via pydantic-settings with env var overrides — extend for new COINBASE_ADV_* vars

### Integration Points
- `CoinbaseClientPool.__init__()` creates two REST clients — will need new credential fields
- `agents/ingestion/main.py` line ~132 creates WS client using config URLs — already correct
- Every agent that calls `client_pool.get_client(target)` — transparent, no changes needed at call sites
- `configs/default.yaml` instruments list — product_id field will be replaced by symbol-based resolution

</code_context>
